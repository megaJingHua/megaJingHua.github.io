提升程式可讀性

2019-06-12
參考資料：如何提升你的程式可讀性之實務技巧

對於軟體開發的設計模式，各自有不同的見解。
但能夠讓程式碼可以在開發時具有「溝通」的效果，卻是一致的目標。

可讀性帶來什麼幫助?

:::warning
難以閱讀的程式碼就像是一座迷宮，不僅迷惑自己還會讓協作者花更多時間去理解。
:::

  1. 提升效率
    通常一段時日過後回頭來修改維護時，會重新挑戰你的記憶力，不易讀懂得程式碼將會再度浪費你的開發時間。所以，可讀性提升的是未來自己和他人的開發效率。

  2. 減少犯錯
    良好的可讀性也就意味著表達明確，當一段程式碼的命名敘述不同於執行內容時，就是叫他媽的誰放的地雷。

    • 命名不明的例子 會以為 increase_read_count 只是單純增加計數，但裏頭偷發一個不知道做什麼用的 Email，這就屬於一種隱諱的寫法，因為和主要敘述功能的名稱不完全符合，也難以聯想。

  3. 幫助他人學習
    能夠讓人更快進入狀況，瞭解其邏輯、設計架構，而減少時間在搞懂這迷宮到底怎麼蓋的，能夠展現其思緒及脈絡。

:::warning
當每個你看到的程式，執行結果都與你想得差不多，你會察覺到你正工作在 Clean Code 上。
—clean code
:::

一.觀念

+ 文字比數字好
+ 一次只做一件事
+ 永遠把自己當新手
+ 填克漏字
+ 勇於展現錯誤
  1. 文字筆數字好
    • 命名不明的例子
    • 命名準確的例子 ::: danger
      注意：可以在程式中以 String 語意化的方式顯示，但在 Database 中使用 integer 可減少儲存空間及提高效率。
      :::
  2. 一次只做一件事
    單一功能原則（Single responsibility principle） 這是物件導向設計五原則 SOLID 的其中一個，大方向是盡可能的保持「單純」。在 function / method 裡，盡可能秉持做一件事而不是一堆事的原則。
    當一個方法有太多的職責時也意味著：耦合多事務、有各種結果及錯誤、不易閱讀。
  • 一個 function 超過一件職責的範例
  • 一個 function 只處理一件事範例 這樣子做有什麼好處？
    function 具有更高的可讀性，如果需要理解實作細節，也可以很容易找到。未來如果要新增「簡訊」通知，是不是我只要在 notify_to_user 裡面添加實作細節就可以了？

  1. 永遠把自己當新手
    永遠把自己當作一個初出茅廬的工程師，以一個閱讀者的角度去思考，能不能快速看懂這些程式碼大概在做些什麼？有多少地方需要解釋才能看的懂？需要解釋的部分是否寫了清楚的註解？而不是把自己當作 compiler，認為只要夠厲害就看的懂。

    :::danger
    高手寫出來的程式能夠讓新手、高手都看的懂。
    偽高手寫出來的程式只有自己能看的懂。
    :::

  2. 填克漏字
    在邏輯或畫面撰寫時，會是先思考這裡程式碼要長怎樣才會好讀，然後在去定義實作方法。

  3. 勇於展現錯誤
    有些程式語言有捕捉例外（Error expection）的語法可以使用，但使用不當很容易變成掩蓋錯誤

    這對調用者來說，他可能會不懂為什麼會被 return，仔細查也沒有 Log 被打印出來。這會讓 debug 變的非常困難，所以建議：

    • 只捕捉自己知道的例外

    • 錯誤要有明確的處理（發通知、寫 Log）

    • 如果不知道怎麼處理，就讓他噴掉，至少還比較容易發現 Bug

      二.命名

      :::warning
      There are only two hard things in Computer Science: cache invalidation and naming things. – Phil Karlton
      :::

    • 化簡為繁

    • 規格不該出現在這

    • 兩個恰恰好

    • 讓包裝取代混亂

  4. 化簡為繁
    不必要的簡寫對閱讀沒有幫助。當你看到 ==ua.lock== 時，你不會明白這是什麼意思。但如果你看到 ==user_account.lock== 就可以第一時間猜到是「使用者帳戶要被鎖定」的操作。

  5. 規格不該出現在這
    不要輕易把規格放在命名裡面，假設有一個方法命名為 ==lock_3_hours==，如果需求突然要改成 5 小時，是不是也要把命名改成 ==lock_5_hours==？
    可以取 ==lock_with_time==，然後把時間用參數傳遞，或是設定預設值==DEFAULT_LOCK_HOUR = 3.hours==。

  6. 兩個恰恰好
    命名的單字使用的越多，通常表示負責的事情也越多。建議保持在 2 個字左右的命名會是比較理想的設計。==saveAndUploadPost==可拆解為==savePost==與==uploadPost==。

  7. 讓包裝取代混亂
    正則表達式的效率會比自己寫方法判斷來的快，但這個東西在閱讀上就不是那麼容易，建議將複雜的表達式寫下註解或是直接用常數處理。

    三.流程

    • Early Return
    • 縮排只要一層
    • 請使用正邏輯

  8. Early Return

    「哦，原來只要沒有錢就會拿到這個字串，那有錢的情況就一定會執行後面的實作了！」

    ::: danger
    注意：不適用於所有程式語言。
    :::

  9. 縮排只有一層
    越深的巢狀我們常戲稱為「波動拳」，有難以閱讀的狀況，因為你很難在視線上能夠對齊整個程式碼邏輯區塊，常常會看到這行不曉得到底在哪一層，然後還要思考這層能夠透過判斷式進來又是什麼條件，所以請盡量保持只有一層的縮排。

  10. 請使用正邏輯
    正邏輯會更容易被閱讀，因為沒有人喜歡看雙重否定句之類的語句，例如「是否不同意不舉行選舉

四.註解

+ 記錄想法，不要寫廢話
+ 使用標籤

  1. 錄想法，不要寫廢話
    註解是幫助理解，不是製造更多困惑。所以要替讀者設想，他讀到這裡可能會不明白為何有這個參數或設定？也許是當時共同決策出來的，請把原因寫清楚方便自己日後考古。

  2. 使用標籤
    很多開發工具可以針對這些通用標籤做處理，日後可以統一快速搜尋相同的標籤來做改善處理。

    • TODO: 代表未實現的功能
    • FIXME: 表示目前有一些 bug 需要修正
    • HACK: 某些原因硬幹出來的
    • XXX: 就是他媽的這段是怎樣，通常代表可以重寫或是沒人敢動了

五.性能 vs 可讀性

::: warning
程式碼是寫給人看的，編譯過後的 0 和 1 才是給電腦看的。
:::
程式碼是寫給人看的不是給電腦看的，只有在編譯過後的 0 和 1 才是真正給電腦看的，所以在與他人協作的情況下，程式可讀性應該為第一重要。許多比較高效能的撰寫方式，通常會以可讀性做為代價，除非你很確定這段程式碼就是效能瓶頸，否則不建議太早最佳化。

六.約定優於配置（convention over configuration）

每個團隊因應成員的組成也有不同的習慣，不見得有一套萬全的方法能夠適用每個團隊。讓協作的隊友能夠有一套方式互相看的懂對方的產出，來達到最大的開發效率，就算有人中途離開了，也不會很難接手他的東西。建立 convention 的過程是需要不斷嘗試並調整。

七.社群規範（Ecosystem）

這部分要端看你所學的程式語言或框架，目前在社群上什麼樣的寫法是比較常見的，盡可能的去遵循。

自立門戶，往往遇到的情況就是：

  • 你公司招的新人，看不懂你的規範，需要花更多時間磨合。
  • 你從公司離開了，到下一間公司，要重新習慣寫法。

八.以測試角度提升可讀性

測試通常有粒度的不同，由小而大，從單元測試到整合測試，也可能會有職責的不同，從工程師負責的 unit test 到 QA 負責的更多整合測試、黑白盒測試等等

撰寫測試的好處是什麼？

  • 替你省掉手動測試的時間
  • 替你省掉回歸測試的時間
  • 提升你的程式碼設計品質

為了要測一個單元，結果必須要製作各種資料才能完成，那就間接的代表，這個 function 耦合了什麼。所以你在設計程式的時候，也可以同時以測試的角度下去看，我這個 function 容易被測試嗎？負責太多事情只會：

  1. 難以閱讀
  2. 難以抓 bug
  3. 難以測試
